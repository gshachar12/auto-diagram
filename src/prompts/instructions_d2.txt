D2 Sequence Diagram Generation Rules (PCAP-Based, Tagged Output Version)

These rules ensure that diagrams generated from PCAP data compile successfully in D2 and can be easily parsed by downstream tools.

The output must contain two sections wrapped in tags:

1. <entities> ... </entities>
2. <diagram> ... </diagram>

The entities section allows other systems to parse network actors, while the diagram section contains the D2 diagram itself.

---

1. Output Format

The final output must follow this exact structure:

<entities>

entity_id | label

entity_id | label

</entities>

<diagram>

D2 DIAGRAM HERE

</diagram>

Example:

<entities>

client_192_168_1_10 | 192.168.1.10 Client
resolver_8_8_8_8 | 8.8.8.8 DNS Resolver

</entities>

<diagram>

shape: sequence_diagram

client_192_168_1_10: "192.168.1.10 Client"
resolver_8_8_8_8: "8.8.8.8 DNS Resolver"

Phase_I: {
client_192_168_1_10 -> resolver_8_8_8_8: "DNS Query example.com"
}

</diagram>

---

2. Entity Extraction From PCAP

Entities represent network actors derived from packet data.

Entities may correspond to:

Client
Attacker
DNS Resolver
Authoritative DNS
Web Server
Container
Router
Load Balancer

Entities should be derived from:

• source IP
• destination IP
• hostname
• protocol role

Example mapping

PCAP packet

192.168.1.10 -> 8.8.8.8 DNS query

Entities

client_192_168_1_10 | 192.168.1.10 Client
dns_google | 8.8.8.8 Google DNS

---

3. Entity Naming Rules

Identifier rules:

• only letters, numbers, and underscores
• no spaces allowed
• must start with a letter

Allowed

resolver
attacker
auth_referral
web_server

Forbidden

Victim Resolver
Auth referral.com

Correct

auth_referral: "Auth referral.com"

---

4. Diagram Structure

The diagram inside the <diagram> tag must follow this structure:

shape: sequence_diagram

entity definitions

phase blocks

edges

Example skeleton:

shape: sequence_diagram

client: "Client"
resolver: "DNS Resolver"

Phase_I: {
client -> resolver: "DNS Query example.com"
}

---

5. PCAP Event → Edge Mapping

Every meaningful network exchange becomes an edge.

Examples

DNS query

client -> resolver: "DNS Query example.com"

DNS response

resolver -> client: "DNS Response example.com A=1.2.3.4"

TCP handshake

client -> server: "TCP SYN"
server -> client: "TCP SYN ACK"

HTTP request

client -> server: "HTTP GET /index.html"

HTTP response

server -> client: "HTTP 200 OK"

Only important protocol events should be included.

---

6. Phase Detection

Large traces should be grouped into phases.

Common phases

Initialization
Handshake
Attack Phase
Amplification
Retry
Failure

Example

Phase_I: {
client -> resolver: "DNS Query xyz.referral.com"
}

---

7. SAFE EDGE RULE (CRITICAL)

Edges in sequence diagrams must remain simple.

Allowed syntax

A -> B: "message"

Example

resolver -> auth_referral: "Resolver queries referral.com"

Do NOT attach style blocks to edges.

Forbidden

A -> B: "message" {
style.stroke: "#FF0000"
}

Forbidden

A -> B: "message"
{
style.stroke: "#FF0000"
}

These patterns cause the error

reserved keywords are prohibited in edges

---

8. Phase Styling

If visual grouping is needed, style the phase block instead of edges.

Example

Phase_I: {
style.fill: "#f0f7ff"

client -> resolver: "DNS Query example.com"
}

---

9. Ghost Edges

Do not generate invisible layout edges because they frequently break D2 sequence diagrams.

---

10. Message Formatting

Messages should summarize the network event clearly.

Examples

"DNS Query xyz.referral.com"
"DNS Referral NS records"
"Delegation response"
"Resolver returns SERVFAIL"
"TCP SYN"
"HTTP GET /index.html"

Avoid raw packet payloads.

---

11. LLM Generation Strategy

To reduce syntax errors when generating diagrams from PCAP:

Step 1
Extract entities from PCAP.

Example

client_192_168_1_10
resolver_8_8_8_8
auth_referral

Step 2
List events in plain text.

Example

1 client queries resolver
2 resolver queries authoritative server
3 authoritative responds with referral
4 resolver performs internal lookups
5 resolver returns failure

Step 3
Convert events into edges.

Example

client -> resolver: "DNS Query xyz.referral.com"
resolver -> auth_referral: "Resolver query referral.com"
auth_referral -> resolver: "Referral response"
resolver -> client: "SERVFAIL"

Step 4
Group edges into phases.

Example

Phase_I: {
client -> resolver: "DNS Query xyz.referral.com"
}

Phase_II: {
resolver -> auth_referral: "Resolver query referral.com"
}

Phase_III: {
resolver -> client: "SERVFAIL"
}

The diagram must only be generated after these internal steps.

---

12. Final Validation Rules

Before output ensure:

• <entities> section exists
• <diagram> section exists
• all entities used in the diagram appear in the entities section
• entity identifiers contain no spaces
• edges follow the format "A -> B: message" only
• no style blocks appear after edges
